生成本质
HUAWEI
AUTOREGRESSIVE GENERATION

大模型不是一次性写答案,而是循环预测下一个字

01
读取上文

输入 prompt 和历史回答共同构成当前上下文,所有后续判断都从这段上下文开始。

02
预测下一个 Token

模型根据上下文分布选择最合理的下一个 token,不是预先保存标准答案。

03
拼回上下文

新 token 立刻成为后续输入的一部分,影响下一步的查询需求和注意力分布。

04
循环直到结束

持续生成、持续回读、持续更新,形成流式输出和长对话的基本运行模式。

底层问题
HUAWEI
THREE PROBLEMS

所有机制都在解决三件小事

01
怎么看懂语义

注意力机制要判断哪些 token 与当前生成最相关,Q/K/V 把这种判断拆成查询、标签和内容三类向量。

对应能力: 指代消解、因果转折、上下文融合、多义词消歧。

02
怎么看懂顺序

Transformer 原生不认识顺序,必须额外注入位置关系;RoPE 用旋转角表达绝对位置和相对距离。

对应能力: 语序、远近、主谓宾、长文本引用。

03
怎么跑得更快

逐字生成会反复读同一段历史,如果每步都重算全文 KV,长上下文推理成本会爆炸。

对应能力: KV Cache、流式输出、长对话响应。

注意力机制
HUAWEI
ATTENTION

注意力的本质: 相似度匹配 + 加权求和

01
打分

当前 token 与所有上文 token 计算相似度,相似度越高,代表语义关联越强。

02
归一化

通过 Softmax 把分数变成权重,让“关注谁、关注多少”变成可计算分布。

03
融合

用权重对上下文语义加权求和,把最相关的信息吸收到当前 token 表示里。

04
输出

得到上下文感知的新向量,再进入后续层继续抽象、组合和预测。

QKV 解释
HUAWEI
QUERY / KEY / VALUE

QKV 是注意力的“需求-标签-内容”检索系统

绝大多数注意力理解卡点,都来自没有把 Q/K/V 的物理意义拆开。

Q
Query 查询向量

当前 token 的需求: 我现在想找什么信息、关注谁、需要哪类上下文。

K
Key 键向量

历史 token 的身份标签: 我是谁、有什么特征、是否值得被当前需求匹配。

V
Value 值向量

标签背后的真实语义内容: 被匹配上之后,真正被吸收和融合的信息。

METAPHOR

Q = 搜索框; K = 网页标题标签; V = 网页正文内容。搜索框匹配标题,再把匹配度最高的正文内容拿回来。

QKV 工作流
HUAWEI
RETRIEVAL LOOP

QKV 的运转顺序决定了注意力如何“检索上下文”

当前 Token 生成 Q

每个新 token 都产生新的查询需求,代表当前一步要寻找的上下文线索。

历史 Token 提供 K

所有历史 token 的标签被拿出来与 Q 匹配,计算谁更相关、谁更重要。

匹配分数变权重

相似度经过归一化变成注意力权重,也就是“每个历史 token 贡献多少”。

加权累加 V

权重越高,V 中语义贡献越大;最终得到融合上文后的新语义表示。

多头注意力
HUAWEI
MULTI-HEAD ATTENTION

多头 = 多组 QKV 并行理解同一段上下文

语序位置

捕捉前后顺序、局部排列和短距离结构。

固定搭配

识别常见词组、术语组合和高频搭配。

代词指代

处理“它、这、其”等跨句引用关系。

远距离依赖

把长文本前段事实带入后段推理。

语法结构

学习主谓宾、修饰、并列、嵌套关系。

逻辑对比

处理因果、转折、递进、条件和让步。

细粒语义

补充情感、语气、隐含态度和语境。

拼接融合

各头结果合并,形成更完整的全局认知。

RoPE 机制
HUAWEI
ROTARY POSITION EMBEDDING

RoPE 把 token 位置转成向量旋转角

向量两两分组

把向量维度拆成二维平面上的若干小组。

按位置旋转

位置越靠后,旋转角度越大。

夹角体现距离

两个 token 的夹角自动体现相对远近。

进入 QK 匹配

位置关系被纳入注意力相似度计算。

RoPE 规则
HUAWEI
POSITION RULES

RoPE 同时解决绝对位置、相对距离和缓存兼容

01
自带绝对位置

旋转角度本身标记 token 是第几个位置,让模型能区分“我爱吃苹果”和“苹果吃我爱”。

02
自带相对距离

Q 与 K 的夹角可反映两者相隔多远,越近越容易形成稳定匹配模式。

03
兼容 KV Cache

历史 K 被旋转后位置固定,缓存后无需反复重算;新 Q 只需按当前位置旋转。

RULE OF THUMB

RoPE 只旋转 Q、K,因为匹配关系需要位置;不旋转 V,因为 V 是纯语义内容,被取回时不需要再携带匹配角度。

KV Cache
HUAWEI
PREFILL TO DECODE

KV Cache 把长上下文推理拆成两阶段

01
Prefill 预填充

首次输入时一次性计算所有上文 token 的 K、V,写入显存缓存;最后一个 token 的 Q 用于启动首个输出。

02
Decode 逐字生成

每生成一个新 token,只计算这个新 token 的 Q/K/V,历史 KV 直接从缓存读取。

03
拼接缓存继续匹配

新 Q 匹配“历史 KV + 新 KV”,得到下一个 token;新 KV 再追加进缓存,循环迭代。

缓存边界
HUAWEI
WHY NOT CACHE Q

为什么缓存 KV,却不缓存 Q?

答案不是工程习惯,而是三类向量的角色不同: 历史标签和内容稳定,当前查询需求持续变化。

KV
可以缓存: 历史上下文已经固定

K 是历史 token 的身份标签,V 是历史 token 的语义内容。上文已经说完,标签和内容不会因为未来生成改变,所以可以长期复用。

价值: 用显存保存历史计算结果,避免每步重算全文上下文。

Q
不能缓存: 当前需求每步都变

Q 代表“我现在想关注谁”。每生成一个新 token,当前语义状态就变了,下一步查询目标也变了,过去 Q 对未来没有复用价值。

结论: 只用新 Q 匹配全局缓存 KV,才是推理加速的核心路径。

完整闭环
HUAWEI
INFERENCE LOOP

Transformer 推理闭环可以压缩成一个中心动作

NEXT
TOKEN
上下文进入模型

prompt、历史输出和新增 token 共同形成当前输入,决定本轮需要关注的信息范围。

注意力读取语义

新 Q 与缓存 K 匹配,按权重吸收 V,并通过多头并行补全语法、指代和逻辑。

RoPE 提供位置关系

Q/K 中的旋转角让模型分清先后、远近和相对位置,避免把词序看成无关排列。

KV Cache 支撑速度

历史 K/V 复用,新 K/V 追加;下一步只需新 Q 继续检索全局缓存。

学习总结
HUAWEI
FINAL FRAMEWORK

大模型复杂技术最终落回三个抽象问题

怎么看懂语义

注意力负责筛选相关 token,QKV 把筛选拆成查询、标签、内容;多头进一步从多个维度并行理解。

怎么看懂顺序

RoPE 把位置写入 Q/K 的旋转角,让模型在语义相似之外,还能感知先后和距离。

怎么跑得更快

KV Cache 把固定历史从重复计算里剥离出来,用空间换时间支撑实时对话和长上下文。

掌握这条链路后,再看模型推理、量化加速、上下文扩展、长文本优化和 agent 工程,就能判断每项技术到底在优化“语义、顺序、速度”中的哪一环。
